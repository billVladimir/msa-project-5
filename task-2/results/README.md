## 2. Обоснование выбора: Kubernetes CronJob

### Почему K8s CronJob

Для данной задачи **Kubernetes CronJob** - оптимальное решение по совокупности факторов:

1. **Простота задачи соответствует простоте решения.** Логика сводится к SQL-запросу с JOIN и записи результата в CSV. Для этого достаточно Python-скрипта в 50–100 строк. Spring Batch, Airflow и тем более Spark вносят неоправданную сложность.

2. **Нативность в инфраструктуре.** Магазин уже работает в Kubernetes. CronJob - стандартный ресурс K8s, не требующий дополнительных компонентов. Не нужно разворачивать и поддерживать отдельный кластер Airflow или Spark.

3. **Минимальная ресурсоёмкость.** Pod запускается строго по расписанию, выполняет работу за секунды (10K строк из PostgreSQL → CSV), освобождает ресурсы. Нет постоянно работающих процессов.

4. **Встроенное CRON-расписание.** `schedule: "0 6 * * *"` - и задача запускается каждое утро в 6:00. Никаких дополнительных планировщиков.

5. **Мониторинг через существующий стек.** Логи Pod доступны через `kubectl logs` и автоматически собираются ELK/Loki. Метрики Job (success/failure) видны в Prometheus через kube-state-metrics.

6. **Retry и отказоустойчивость из коробки.** `backoffLimit`, `activeDeadlineSeconds`, `restartPolicy` обеспечивают повторные попытки при сбоях.

### Почему НЕ другие решения

| Решение            | Причина отклонения                                                                                                                                                            |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Spring Batch**   | Избыточная абстракция (Reader/Processor/Writer) для задачи без трансформации данных. Требует JVM, увеличивает потребление ресурсов. Оправдан при сложной пошаговой обработке. |
| **Apache Airflow** | Требует развёртывания целого кластера (Scheduler, Webserver, Workers, DB) ради одного простого задания. Оправдан при десятках взаимосвязанных пайплайнов.                     |
| **Spark**          | Предназначен для обработки терабайтов данных в распределённом режиме. Для 10K строк - кратное превышение необходимых ресурсов и сложности.                                    |

## 3. C4-диаграмма контекста (To Be)

Диаграмма находится в файле [c4_context.puml](c4_context.puml).

### Описание работы решения

1. **Kubernetes CronJob** срабатывает каждый день в 6:00 UTC по расписанию `0 6 * * *`.
2. K8s создаёт **Pod** с контейнером `price-list-exporter` (лёгкий Python-образ).
3. Скрипт подключается к **PostgreSQL** (БД прайс-листов) и выполняет SQL-запрос, объединяющий таблицы `products`, `categories`, `clients` и `client_prices`.
4. Для каждого B2B-клиента формируется индивидуальный **CSV-файл** с ценами.
5. Готовые файлы загружаются в **S3-совместимое хранилище** (MinIO / AWS S3 / GCS), откуда B2B-клиенты могут их забрать или получить ссылку на скачивание.
6. При успешном/неуспешном завершении скрипт отправляет метрику в **Prometheus** (через Pushgateway), а логи собираются **ELK-стеком**.
7. Pod завершается и уничтожается - ресурсы освобождаются.
